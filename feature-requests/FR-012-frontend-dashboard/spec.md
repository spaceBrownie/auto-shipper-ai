# FR-012: Frontend Dashboard

## Problem Statement

The commerce engine operates autonomously — scheduled jobs run cost gates, stress tests, margin sweeps, SLA checks, and kill rules without human intervention. But the operator still needs real-time visibility into what the system is doing and why. Without a dashboard, the engine is a black box: margins can degrade, vendors can underperform, SKUs can be terminated, and the operator wouldn't know until reviewing logs.

The frontend must surface the data generated by the backend in a clear, actionable interface that matches the solo-operator spec's 8 core views (Section 7.1), plus a compliance view backed by the existing audit trail.

## Business Requirements

### Core Views (solo-operator spec Section 7.1)

1. **SKU Portfolio** — All active SKUs with real-time margin, state, and signals. Actions: pause, terminate, scale.
2. **Cost Gate Runner** — Input a product idea; run the full cost gate (13-component verification) + stress test. Actions: approve to launch, reject to kill.
3. **Vendor Scorecard** — Vendors with SLA performance, reliability scores, and breach history. Actions: pause vendor, view SKU impact.
4. **Margin Monitor** — 90-day rolling margin chart per SKU and total portfolio. Drill into component breakdown.
5. **Experiment Tracker** — Active listing hypotheses with validation window countdown and verdict. Actions: kill or scale. No budget concept — zero-capital model.
6. **Capital Overview** — Reserve balance, 30-day revenue, SKU-level P&L. View reserve health.
7. **Demand Signals** — Latest `DemandScanJob` output — trending categories. Actions: initiate new product validation. (Placeholder until DemandScanJob is implemented.)
8. **Kill Log** — History of auto-terminated and paused SKUs with termination reasons. Learn from patterns.

### Additional View

9. **Compliance Status** — Per-SKU compliance audit trail showing IP, claims, processor, and sourcing check results. View latest result and full history.

### Approval Flows (write operations)

- **Cost Gate Runner**: trigger cost verification and stress test for a SKU
- **Experiment lifecycle**: create experiment, mark validated (link to SKU), mark failed
- **Kill confirmation**: confirm kill recommendations from KillWindowMonitor
- **SKU state transitions**: pause, terminate, scale decisions
- **Vendor actions**: register vendor, update checklist, activate, compute score

### Data Model Alignment (zero-capital model, spec v1.2)

- No "budget" concept for experiments — experiments track validation window and estimated margin per unit
- No "capital deployed" metric — the system is zero-capital per product; customer payment covers all costs
- "Capital reallocation" is replaced by **priority ranking** — ranking SKUs by risk-adjusted return
- Portfolio summary shows: total experiments, active experiments, active SKUs, terminated SKUs, blended net margin, total profit
- Systemic refund alerts surface portfolio-wide patterns (3+ SKUs > 3% refund rate in same 7-day window)

## Success Criteria

- React + Vite + shadcn/ui + TypeScript project initialized and building
- All 9 views render live data from the Spring Boot backend REST API
- SKU Portfolio page renders from `GET /api/skus` with state filter
- SKU detail page with cost breakdown (`GET /api/skus/{id}` + cost envelope), stress test result, pricing history (`GET /api/skus/{id}/pricing`), P&L (`GET /api/capital/skus/{id}/pnl`)
- Cost Gate Runner page triggers `POST /api/skus/{id}/verify-costs` and `POST /api/skus/{id}/stress-test`
- Vendor Scorecard renders from `GET /api/vendors` with reliability scores (`POST /api/vendors/{id}/score`)
- Margin Monitor renders margin trend charts from margin history endpoint
- Experiment Tracker renders from `GET /api/portfolio/experiments` with validation window countdown
- Capital Overview renders reserve health from `GET /api/capital/reserve`
- Demand Signals page shows placeholder UI (backend not yet implemented)
- Kill Log renders terminated SKUs and confirmed kill recommendations
- Compliance Status renders from `GET /api/compliance/skus/{id}`
- Priority ranking renders from `GET /api/portfolio/reallocation`
- Refund alerts render from `GET /api/portfolio/refund-alerts`
- All pages handle loading, error, and empty states
- Deployed and accessible at `http://localhost:5173` in development

## Non-Functional Requirements

- TypeScript strict mode enabled
- React Query (TanStack Query) for all data fetching and caching
- Recharts for margin trend and chart visualization
- shadcn/ui for all UI components (tables, cards, badges, forms, gauges)
- Responsive layout — usable on 1280px+ screens
- No authentication in Phase 1 (internal tool, single operator)
- Served at `localhost:3000` in production (Docker), `localhost:5173` in development (Vite)

## Backend API Gaps (must be addressed during implementation)

Two new REST endpoints are needed to fully support the dashboard:

1. **Margin history endpoint** — `GET /api/capital/skus/{id}/margin-history?from={date}&to={date}`
   - Backed by `MarginSnapshotRepository.findBySkuIdAndSnapshotDateBetweenOrderBySnapshotDateAsc()`
   - Returns daily margin snapshots for chart rendering
   - Response: list of `{ snapshotDate, grossMarginPercent, netMarginPercent, refundRate, chargebackRate }`

2. **SKU state history endpoint** — `GET /api/skus/{id}/state-history`
   - Backed by `SkuStateHistoryRepository`
   - Returns state transitions for kill log and SKU detail timeline
   - Response: list of `{ fromState, toState, transitionedAt }`
